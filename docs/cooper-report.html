Cooper — Realtime Voice Agent: Capabilities & Hardening Report

An ambient AI co-pilot for executive calls — live transcription, in-conversation tools, and a post-call work pipeline that turns meetings into PRDs, plans, and prototypes. This report assesses what Cooper does well today and what it needs before production.

01 · Overview

What Cooper is, and what it does

Cooper is a progressive web app that joins executive calls as an ambient assistant. It connects to OpenAI's Realtime API (v2) over WebRTC for bidirectional voice, transcribes the conversation live, can invoke tools mid-conversation, and after the call uses the Responses API to synthesize structured artifacts — all behind password-gated access and silent until summoned by a wake phrase.

🎙️

Live voice & transcription

Bidirectional WebRTC audio with semantic VAD, far-field noise reduction, and live transcription via gpt-4o-mini-transcribe, with deduplicated transcript capture.

🤫

Silence-by-default UX

Cooper stays quiet during meetings and only engages on a wake phrase ("Hey Cooper", "What do you think, Cooper") — designed for ambient, non-intrusive presence.

🧰

In-conversation tools

Seven Realtime tools: check_calendar, search_workspace_context, get_customer_context, inspect_engineering_context, create_followup_action, run_gstack_skill, create_canvas_artifact.

📦

Post-call work pipeline

A rate-limited background queue turns transcripts into 8 artifact types — post-call kit, execution plan, PRD, code sketch, HTML prototype, Mermaid diagram, UI wireframe, follow-up summary.

🖼️

Sandboxed artifact preview

Markdown rendered through DOMPurify; HTML prototypes displayed in a sandboxed iframe with mobile/desktop toggle. Mermaid diagrams supported in markdown view.

🗂️

Project context system

Ingests pasted text, Markdown, and PDF uploads, compacts them into a bounded context packet, and injects them into the live session and post-call prompts.

🧠

GStack advisory skills

Read-only advisory reviews (CEO, engineering, code, QA, design, spec, office hours) executed server-side via the Responses API — never mutating code or deploying.

🔌

Arcade integrations

Workspace/customer/engineering context lookups and write actions through the Arcade SDK, behind an OAuth pre-authorization flow and explicit write-confirmation gates.

📲

Installable PWA

Service-worker offline shell, installable manifest, and push notifications for completed post-call work — a hybrid SSE + polling state model keeps the UI live.

02 · Architecture

How it works

A single Node.js process serves both the React SPA and an Express API. The browser holds the WebRTC media connection directly to OpenAI; the server relays the SDP handshake (keeping the API key server-side), runs the tool layer, and drives the artifact job queue.

Client · PWA

React 19 SPA

Lock screen, call canvas, artifact preview, project context UI. Establishes the WebRTC peer connection and oai-events data channel.

Edge

Express server.js

Auth middleware on all /api & /session routes. Serves SPA (Vite dev middleware or built dist/). SSE broadcast on /api/events.

External

OpenAI Realtime v2

gpt-realtime-2 over WebRTC. Server relays SDP to /v1/realtime/calls with the Bearer key; audio flows browser ↔ OpenAI directly.

request & event flow
Security

Auth & sessions

Password gate with timing-safe compare; HMAC-SHA256 signed HTTP-only cookies (SameSite=Lax, Secure in prod), 168h TTL.

Compute

Tool execution layer

Routes Realtime function calls to local handlers, GStack skills, canvas jobs, or the Arcade proxy. Risk-tagged; writes gated.

Compute

Artifact job queue

In-process worker polling every ~15s, multi-step Responses API calls with retry/backoff, writes files to data/artifacts/.

persistence & integrations
State

JSON data store

data/cooper.json holds calls, transcripts, artifacts, jobs, tool calls, projects, and Arcade authorizations via a serialized write queue.

External

OpenAI Responses API

Post-call artifact synthesis and GStack advisory reviews; reasoning-effort tuned, output-token capped.

External

Arcade SDK

OAuth pre-authorization then proxied tool execution for workspace, customer, and engineering context and write actions.

03 · What's Good

Genuine strengths worth celebrating

Cooper is a thoughtfully built product, not a prototype. Several decisions reflect real engineering maturity across security, UX, and operations.

✓ Clean WebRTC integration

The browser holds the media connection directly to OpenAI while the server relays only the SDP handshake — keeping latency low and the API key server-side. Robust connection-error handling covers getUserMedia, peer-connection, and SDP-exchange failures with user-friendly messages.

✓ Server-side key handling

OPENAI_API_KEY never reaches the browser — all Realtime and Responses calls are made server-side with the Bearer token. The SDP relay is the right pattern for keeping secrets off the client.

✓ Silence-by-default ambient UX

Cooper stays quiet during meetings and only responds on wake-phrase detection — a deliberate, respectful design for an executive co-pilot that earns its place in the room.

✓ Sandboxed artifact preview

HTML prototypes render in an iframe via srcDoc with a restrictive sandbox; markdown is sanitized with DOMPurify before injection. Generated content is isolated from the host app.

✓ Strong session cryptography

HMAC-SHA256 signed tokens (base64url JSON + signature) with timing-safe comparison, HttpOnly + SameSite=Lax cookies, Secure flag in production, and configurable TTL. Password checks use constant-time comparison.

✓ The post-call work pipeline

A rate-limited, retrying background queue (respecting retry-after headers, exponential backoff) turns transcripts into eight artifact types via multi-step Responses calls — a genuinely useful workflow that runs without blocking the call.

✓ Write-action safety gates

Tools are risk-classified (read / advisory / write). Writes are disabled by default (COOPER_ENABLE_ARCADE_WRITES=false) and require explicit confirmation, and GStack skills are advisory-only — they cannot mutate files, deploy, or open PRs.

✓ Secrets redaction & audit trail

Sensitive-text detection (API keys, tokens, private keys, bearer tokens) redacts before logging, and every tool call is recorded with timestamp, status, risk level, and duration for an auditable history.

✓ Installable PWA + live state

Offline shell caching, installable manifest, and completion notifications, backed by a resilient SSE + 4s-polling hybrid that keeps the UI consistent even if the event stream drops.

✓ Bounded context & job recovery

Project sources are truncated (≤250k per source, ≤18k per session) to contain prompt bloat, and running jobs are requeued with a log entry on restart — pragmatic resilience for a single-process design.

04 · Where It Needs Hardening

Prioritized gaps before production

Findings span security, reliability, scalability, observability, and testing. Each card carries a severity and a concrete recommendation. Several gaps are amplified by Cooper's single-tenant, single-process design.

Critical · 1 High · 9 Medium · 13 Low · 9

Critical

Critical · Security

API keys present in .env / git working tree

OPENAI_API_KEY and ARCADE_API_KEY appear in the .env file and likely the commit history, used directly in fetch headers without explicit masking in error paths.

Fix: Rotate the OpenAI and Arcade keys immediately. Move secrets to a managed store (Vault / AWS SSM / Railway), confirm .env is gitignored and scrub history, and add secret-scanning to CI. Never log raw env values or full request headers.
.env · server.js:228,251,948,960

High

High · Security

No brute-force protection on login

POST /api/auth/login accepts unlimited password guesses against a single shared password with no rate limiting, lockout, or audit logging.

Fix: Add sliding-window rate limiting (e.g. 5 attempts / 15 min / IP) with exponential backoff and lockout. Log failed attempts with IP. Hash the password (bcrypt) rather than comparing plaintext.
server.js:63-84, 2088-2095
High · Security

No tenant / ownership isolation

Any authenticated user can read and edit all projects, calls, artifacts, and transcripts; endpoints assume "authenticated = authorized" with no row-level checks.

Fix: Add a userId/tenant field to calls, projects, and artifacts; validate ownership on every GET/PATCH/POST and scope /api/state to the current user.
server.js:281-334, 540-553
High · Security

Credentials stored unencrypted at rest

Arcade authorization tokens and ingested project sources (customer info, PRDs, code) are stored in plaintext in data/cooper.json — exposed if the file leaks.

Fix: Encrypt sensitive fields at rest (libsodium/TweetNaCl) with a key held outside the database, decrypting only when needed for API calls.
server.js:1334-1357, 1575-1593
High · Security

API key exposure risk in errors/logs

Raw OpenAI error payloads (revealing model names and endpoint structure) and unmasked Bearer tokens may surface in error responses or logs.

Fix: Return generic client errors; log full detail server-side only. Mask Bearer tokens, and log method/URL rather than full headers.
server.js:248-274, 957-992, 979-983
High · Security

Prompt injection via untrusted transcript & context

Transcript text and user-pasted project context are concatenated into Realtime instructions and post-call prompts with minimal isolation — an injection vector ("ignore previous instructions…").

Fix: Wrap untrusted content in explicit delimiters (e.g. [TRANSCRIPT_START]…[TRANSCRIPT_END]), label it untrusted, strip control characters, and keep it out of the system-instruction slot.
server.js:145-152, 994-1037 · src/main.jsx:793,830
High · Security

Session secret defaults to app password

COOPER_SESSION_SECRET falls back to COOPER_APP_PASSWORD when unset, making session signing only as strong as the login password.

Fix: Require an independent COOPER_SESSION_SECRET; fail startup if missing or identical to the password. Generate via crypto.randomBytes(32).
server.js:32
High · Security

XSS surface in rendered artifacts

Markdown renders via dangerouslySetInnerHTML; DOMPurify config is minimal and whitelists target/rel, enabling tabnabbing and leaving the configuration unverified against common payloads.

Fix: Harden the DOMPurify config, force rel="noopener noreferrer" on links, add a Content-Security-Policy, and test with onerror/svg/script payloads.
src/main.jsx:2102, 2329-2331
High · Reliability

Single point of failure in data store

All state lives in one JSON file with no backup or replication; concurrent writes rely on unverified fs.writeFile atomicity, risking corruption.

Fix: Migrate to a transactional store (SQLite/Postgres) or add atomic temp-file-rename writes plus scheduled backups. Verify the write-queue under load.
server.js:1503-1534, 19-21
High · Reliability

No process supervision or health checks

A crash takes the whole app down with no auto-restart, and there is no health endpoint for an orchestrator to detect degraded state.

Fix: Run under a supervisor (systemd / PM2 / container restart policy) and add a /healthz endpoint reporting queue depth and last-write status.
server.js:1-2239

Medium

Medium · Security

No CSRF protection on mutations

State-changing POST/PATCH endpoints rely solely on SameSite=Lax cookies; a same-site XSS could forge requests.

Fix: Add CSRF tokens (double-submit cookie or session nonce) and/or strict Origin-header validation on all mutations.
server.js:281,555,638,658,687
Medium · Security

iframe sandbox permits scripts & popups

sandbox="allow-scripts allow-popups allow-modals" lets prototype HTML run JS and exfiltrate via postMessage/window.open.

Fix: Drop allow-modals/allow-popups, add a CSP meta in generated HTML, and validate output against external fetch/embed/object tags. Add a parent-side postMessage origin check.
src/main.jsx:2157-2162
Medium · Security

Path-traversal pattern in artifact file reads

artifactFileName() derives a path from artifact.file via split().pop(); a poisoned field could escape data/artifacts/.

Fix: Construct paths only from artifact.id + a validated extension enum; never trust stored file paths. Use path.basename and a strict regex.
server.js:1962-1967, 680
Medium · Security

SSE broadcast lacks per-client filtering

/api/events pushes all job/state events to every connected client — cross-user leakage in a shared deployment.

Fix: Key the eventClients map by userId and filter broadcasts, or scope events to the owning user.
server.js:525-538, 2146-2151
Medium · Security

No explicit CORS configuration

No CORS middleware; same-origin assumption breaks if frontend and backend split, and defaults are permissive.

Fix: Add restrictive CORS with an ALLOWED_ORIGINS allowlist, credentials:true, and a limited method/header set.
server.js:1-60
Medium · Security

Service worker caches sensitive HTML

The SW caches non-API responses (potentially sensitive shell HTML) with no cache invalidation on logout and a never-incremented cache version.

Fix: Clear caches on logout, version cache names per release, and set Cache-Control: no-store on sensitive routes.
public/sw.js:16-30
Medium · Reliability

No transactional recovery mid-artifact

A Responses failure partway through a multi-step job marks it failed with manual retry only; no transactional guarantees if the process dies mid-step.

Fix: Persist per-step progress so jobs resume from the last completed step, with automatic bounded retries.
server.js:768-945
Medium · Scalability

No API / token rate limiting per user

Only the job queue is throttled; nothing caps per-user artifact requests or OpenAI token spend, allowing quota exhaustion / DoS via queue flooding.

Fix: Add per-user/IP request limits, a queue-depth cap (429 beyond N), and a daily token budget per user.
server.js:661-669, 768-803
Medium · Observability

Tool audit redaction is partial

safeAuditObject redacts only known key names; free-text fields (query, input, context, prompt) may carry PII or secrets into stored tool calls.

Fix: Redact values matching sensitive patterns, truncate strings, and store full responses separately under encryption with retention limits.
server.js:468-523, 1899-1944
Medium · Reliability

Arcade calls not retried; token expiry unhandled

Arcade tool execution fails immediately on transient network errors, and auth-token expiry is not handled gracefully.

Fix: Add bounded retry with backoff for Arcade calls and detect/re-trigger the authorization flow on expiry.
server.js:387-1297
Medium · Scalability

Cannot scale horizontally

Single-threaded event loop, in-process queue, and local file DB mean no multi-process or multi-node operation without data migration and distributed locking.

Fix: Externalize the queue (Redis/BullMQ) and DB; add distributed locking before scaling beyond one instance.
server.js (architecture)
Medium · Reliability

Context limits enforced too late

Upload size/format is validated only when building context; a 20MB file is accepted then silently truncated later, and PDFs are parsed fully in memory.

Fix: Validate size/format and magic bytes at upload, stream or cap PDF parsing, and surface truncation to the user.
server.js:359-385, 1596-1635
Medium · UX

UI unlock state outlives the session

The localStorage "cooper.entered" flag persists after the server cookie expires; the UI keeps showing as unlocked until a 401 is caught on the next poll.

Fix: On any 401 from /api/state, clear the flag and redirect to the lock screen immediately; re-auth SSE on reconnect.
src/main.jsx:99, 194-196

Low · Security

Weak artifact format validation

Artifact serving checks MIME/extension but not actual file format; no X-Content-Type-Options or Content-Disposition headers.

Fix: Validate magic bytes, add nosniff and Content-Disposition headers, and apply a CSP to artifact routes.
server.js:671-685
Low · Reliability

Unbounded transcript entry size

POST /api/calls/:id/transcript accepts arbitrary text with no per-entry or per-call cap, allowing unbounded DB growth.

Fix: Cap individual entries (e.g. 10k chars) and total call transcript size; enforce a request body limit.
server.js:610-636
Low · Security

Cookies lack Secure flag in dev

The Secure flag is only set in production; acceptable for localhost but worth documenting.

Fix: Gate behind an INSECURE_DEV flag and document the localhost-only assumption in the README.
server.js:76-82
Low · Security

File upload validated by MIME only

Uploads are accepted on client-supplied MIME/extension without magic-byte verification, so spoofed files can pass.

Fix: Verify magic bytes (file-type), reject non-%PDF PDFs, and add per-page extraction timeouts to resist zip bombs.
server.js:359-385, 1596-1635
Low · Observability

Minimal structured logging

Only a handful of console statements; no request logging, duration metrics, or persisted stack traces — debugging production issues is hard.

Fix: Adopt a structured logger (pino) with request IDs, durations, and error stacks, shipped to log aggregation.
server.js (logging)
Low · Testing

No automated test coverage observed

No unit/integration tests guard the auth, tool routing, artifact pipeline, or injection-handling logic — regressions can ship silently.

Fix: Add tests for auth/session signing, tool routing and write gates, prompt-construction escaping, and artifact path safety; wire into CI.
repo-wide
Low · Security

Error messages leak internal details

Client error responses include raw model names and OpenAI API structure.

Fix: Map to generic client-facing messages; retain detail in server logs only.

Low · Reliability

No timeout on getUserMedia

Mic-permission requests use the browser default; a dismissed prompt can hang call initiation.

Fix: Wrap getUserMedia in a Promise.race with an explicit timeout and a clear fallback message.

Silent failures & missing load states

Malformed Realtime events fail silently, and artifact-content fetch errors show stale content with no loading/error UI.

Fix: Surface parse/fetch errors in the UI, add explicit loading and error states, and log the underlying failure.

05 · Production Readiness Scorecard

Where Cooper stands today

Ratings reflect current state against production expectations. Cooper's UX and core feature engineering are strong; the gap to production is concentrated in security hardening, reliability, and scale.

Overall: strong product, pre-production engineering

Cooper is a genuinely well-designed assistant with mature UX and a useful work pipeline. It is production-ready for a trusted single user on a private host today, but multi-user or public deployment requires the Phase 1 and 2 hardening below — starting with key rotation.

06 · Roadmap

Next steps to production

A phased path: stop the bleeding, then harden for multi-user, then scale and operationalize.

Phase 1

Stop the bleeding

Now · days
  • Rotate OpenAI + Arcade keys; move to a secret manager; scrub git history and add CI secret scanning.
  • Require an independent session secret; fail startup otherwise.
  • Add login rate limiting, lockout, and auth audit logging.
  • Mask keys and generic-ize error responses; log detail server-side only.
  • Run under a process supervisor; add a /healthz endpoint.
Phase 2

Harden for multi-user

Near-term · weeks
  • Add tenant/ownership isolation across calls, projects, artifacts, and SSE.
  • Encrypt credentials and project sources at rest.
  • Wrap untrusted transcript/context in delimiters; harden DOMPurify + iframe sandbox + CSP.
  • Add CSRF tokens and restrictive CORS.
  • Migrate the JSON store to SQLite/Postgres with backups and atomic writes.
  • Stand up an automated test suite in CI.
Phase 3

Scale & operationalize

Strategic · quarter
  • Externalize the job queue (Redis/BullMQ) with distributed locking; enable horizontal scaling.
  • Add per-user rate limits and token budgets; per-step job recovery and Arcade retries.
  • Adopt structured logging, metrics, tracing, and log aggregation.
  • Validate uploads by magic bytes; enforce context limits at ingest.
  • Publish deployment, backup, and security runbooks.
